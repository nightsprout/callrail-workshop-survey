1. The How AI Works

If you're coming from ChatGPT, one thing might surprise you: how much everything in Claude — and AI engineering broadly — revolves around tokens. Token limits. Token costs. Context tokens. Output tokens. The word comes up constantly because tokens are the fundamental unit of everything happening under the hood. Understanding them at an engineering level makes everything else click.

It starts with tokens

AI doesn't read words the way you do. It breaks everything down into tokens — fragments of text, roughly three-quarters of a word each. The word "development" is 2 tokens. "CallRail" is 2 tokens. A short paragraph might be 50 tokens. A full source file could be thousands.

Token — The smallest unit of text AI processes. Roughly ¾ of a word (~4 characters). Everything you send, everything AI reads, everything it generates — all measured in tokens.
Advanced: How tokenization actually works

Tokens are subword units — a deliberate engineering tradeoff. If you've ever thought about text encoding or compression, the reasoning will feel familiar.

ApproachVocabulary sizeTradeoff
Characters~256 entriesTiny vocabulary, but "function" takes 8 tokens. Sequences get unmanageably long — like reading letter-by-letter.
Whole wordsMillions"run", "runs", "running", "runner" each need their own entry. The dictionary explodes and can't handle words it's never seen.
Subwords~100K entriesThe sweet spot. Common words like "the" are one token. Less common words split: "tokenization" → "token" + "ization". Handles any text.

Modern LLMs use Byte Pair Encoding (BPE) — an algorithm that scans a massive training corpus, finds the most frequent character pairs, and iteratively merges them into single tokens. Common patterns get compact representations; rare strings decompose into smaller known pieces.

A few things useful to know as an engineer:

  • Code tokenizes differently than prose. Keywords like function and return are often single tokens. Variable names get split. Whitespace and indentation cost tokens too — one reason minified code is cheaper to process.
  • Non-English text burns more tokens. English words get the most efficient encodings. CJK characters, Cyrillic, and emoji all cost more tokens for the same semantic content.
  • Tokenization is why AI miscounts. It doesn't "see" individual letters — it sees subword chunks. This is why it sometimes struggles with character-level tasks like counting letters in a word.

Tokens matter because they're what you're paying for. Every file Claude reads, every command output it processes, every response it writes — all tokens. A long conversation isn't just slower, it's literally more expensive per message, because the entire conversation history rides along with every new message you send.

AI mostly starts blank every time

This is the single most important thing to understand. Claude doesn't remember the details of your last conversation. It doesn't know what you worked on yesterday. Every new session starts mostly from zero.

The exception: Claude Code keeps a lightweight memory across sessions — things like your preferences, project patterns it's noticed, and corrections you've made. These get automatically injected into new conversations so you don't have to repeat yourself on the basics. But this memory is thin — it's not recalling your full conversation history, just key facts it decided were worth remembering.

Beyond that, everything Claude knows in a given conversation comes from what's been injected into that conversation's context window — the working memory for that session.

Context window — The total amount of text AI can hold in working memory for a single conversation. Claude's is 200,000 tokens today, with 1,000,000-token windows already announced. Bigger isn't always better — what matters is what's in the window.

Your prompt, the conversation history, any files Claude reads, any command outputs — it all lives in the context window. When it fills up, older content starts getting compressed or dropped. And here's the catch: AI is best at remembering what's at the beginning and end of the context window. Stuff in the middle gets hazy — researchers call this the "lost in the middle" problem.

Context — Everything the AI can "see" during a conversation: your messages, its responses, files it's read, command outputs. More context isn't always better — focused, relevant context beats a massive dump.
Advanced: Why the middle gets lost

LLMs use an attention mechanism — each token "attends to" every other token in the context window to decide what's relevant. In theory, attention can reach anywhere. In practice, models develop a strong bias toward the beginning (system prompt, initial instructions) and end (most recent messages) of the context.

This has real implications for how you work:

  • CLAUDE.md loads at the top — it's in the highest-attention zone, which is why it's so effective for persistent instructions.
  • Your latest message is at the bottom — also high attention. This is why restating your goal in follow-up messages helps.
  • Message 15 of a 30-message conversation? That's the dead zone. Critical details from early-middle conversation turns get effectively forgotten.

This is also why /clear and starting fresh is so powerful — it's not just about token cost, it's about moving important context back into the high-attention zones.

Generation is prediction

When Claude writes a response, it's not retrieving answers from a database. It's predicting the next token, one at a time, based on patterns learned during training. This is why the same prompt can produce different outputs — it's probabilistic, not deterministic. It's also why giving Claude better context produces better outputs. More signal, less noise, better predictions.

That's the whole loop: tokens in → context window → prediction out. Everything else we'll cover — managing context, CLAUDE.md, plan mode, steering — is about optimizing this loop.

Advanced: Temperature and why outputs vary

At each step, the model doesn't pick one "right" next token — it generates a probability distribution across its entire vocabulary. The word after "the dog sat on the" might be "mat" (35%), "floor" (20%), "couch" (12%), "ground" (8%), and so on.

Temperature controls how this distribution gets sampled. Low temperature (closer to 0) makes the model pick the highest-probability token almost every time — more deterministic, more conservative. High temperature spreads the probability more evenly — more creative, more unpredictable, more likely to hallucinate.

Claude Code uses carefully tuned temperature settings you don't need to adjust. But understanding this helps explain why the same prompt can produce different code on different runs — and why "regenerate" sometimes gives you a better answer.

Chat is what you'd expect — the same Claude from claude.ai, but running natively on your machine. Double-tap Option (Mac) to pull it up over any window for quick questions.

Cowork is for work that takes real effort. Give Claude a folder and a task — "research these three competitors and write a summary" or "analyze this contract and flag risks" — and it reads what's there, works through it, and saves finished output back. It can even run on a schedule for recurring tasks like daily briefings. Think of it as a capable research assistant with access to your files.

Code is the same Claude Code engine that runs in your terminal, but inside the desktop app with visual diffs and a built-in terminal. It has three interaction sub-modes:

  • Ask: Claude proposes every change with a visual diff — you review and approve before anything is modified
  • Code: Changes apply automatically, but terminal commands still require approval
  • Plan: Claude outlines a strategy before executing anything (same concept as plan mode in the CLI)

The important thing to understand: Cowork and Code run on the same engine underneath. Both can work independently, spin up sub-agents for parallel tasks, and sustain long-running work. The difference is scope — Code has your full project and terminal, Cowork is limited to the folders you share.

In the workshop, we'll focus on Claude Code (the CLI), but everything you learn about prompting, context management, and CLAUDE.md applies across all three modes.

Modalities: beyond text

A modality is a type of data a model can work with — text, images, audio, video. Early LLMs were text-only: text in, text out. Modern models are increasingly multimodal, meaning they can process (and sometimes generate) multiple types of data natively.

This matters practically because it determines what you can do with a model. Can you paste a screenshot of a bug and ask "what's wrong?" Can you feed it a design mockup and say "implement this?" Can you have it listen to a meeting recording?

Multimodal — A model that can process more than just text. Claude can analyze images, PDFs, and code alongside text. Other models add audio and video input, or generate images natively. More modalities = more ways to interact.

3. Claude Code & Other Agentic Coders

You may have used GitHub Copilot, or ChatGPT, or even claude.ai for coding questions. Claude Code — and agentic coders like it — are fundamentally different from all of these. Understanding how they're different is the key to using them well.

The spectrum

Think of AI coding tools on a spectrum:

Autocomplete → Chat (copy/paste) → Inline Editing → Agentic

Autocomplete suggests the next line. Chat tools answer questions you copy back into your editor. Inline editors suggest changes in context. Agentic tools do the work.

Agentic — An AI that doesn't just answer questions — it takes actions. It reads your files, runs commands, creates branches, writes code, runs tests, and iterates on its own work. You describe what you want; it figures out how to get there.

Claude Code sits at the far end of that spectrum. It's not suggesting code in your editor — it's operating your entire development environment. It can:

  • Read and write files across your whole project
  • Run shell commands (build, test, lint, deploy)
  • Search your codebase for patterns and dependencies
  • Create branches, make commits, open pull requests
  • Plan multi-step changes before executing them
  • Spawn sub-agents to research or work in parallel

It asks permission before making changes (by default), so you're always in control. But the scope of what it can do is fundamentally different from autocomplete — it's closer to a junior developer with perfect memory of your codebase than it is to a fancy text completer.

Where it lives

Claude Code runs in your terminal. Not in a browser tab, not in an IDE sidebar — in the same terminal where you run git, rails, npm, and everything else. This is intentional: it has access to the same tools you do.

There's also a desktop app for more visual conversations, and integrations with GitHub (for PR review) and Slack (for team use). But the core experience is the CLI.

CLI — Command Line Interface. Claude Code runs in your terminal alongside your other dev tools. This gives it access to your filesystem, git, and every CLI tool you have installed.

4. Where to Start

You don't need advanced prompting skills to get value from Claude. These four use cases work well right out of the box — they're low-friction, immediately useful, and a great way to build intuition before the workshop.

Generate documentation

Point Claude at a module, service, or API and ask it to write docs. It reads the code, understands the structure, and produces documentation that's surprisingly good as a first draft. You can also ask it to update existing docs when code changes — or add a rule to CLAUDE.md so it happens automatically.

Write tests

Claude is excellent at generating test coverage for existing code. It doesn't just write happy-path tests — it finds edge cases, error conditions, and boundary scenarios you might not think of. Point it at an untested module and ask for comprehensive specs. It's also great for TDD: describe what you want to build and ask for failing tests first.

Create demo and seed data

Need realistic fake data for a prototype, a demo, or a dev database? Describe the shape of the data you need and any constraints (realistic names, seasonal distribution, proper phone number formats) and Claude generates it. What used to take an afternoon of Faker scripting takes a single prompt.

Learn and explore

Claude is a patient teacher. Ask it to walk you through an unfamiliar codebase, explain a design pattern, or break down how a tool works — and it'll explain things in terms of your own stack. This is one of the most underrated use cases: using Claude as an always-available mentor who never gets tired of questions.

5. Talking to Claude

Good news: you don't need to learn a special language. Most of the time, you talk to Claude the way you'd talk to a colleague. "Fix the failing test in user_spec.rb." "Add pagination to the trackers endpoint." "What does this service do?"

That said, when you're not getting the results you want, there's a formula that helps.

The prompting formula

ComponentWhat it doesExample
GoalWhat you're trying to accomplish"I need to add rate limiting to our API"
RoleWhat perspective Claude should take"Act as a senior Rails engineer familiar with our stack"
ProblemWhat's broken, missing, or needed"We're getting hammered by a single client doing 1000 req/min"
DetailsConstraints and context"We use Redis, Rails 7, and rack-attack is already in the Gemfile"
ExamplesWhat good output looks like"Similar to how we rate-limit the webhooks controller"
Prompt — The message you send to AI. Can be a simple question or a detailed request. Better prompts produce better results, but don't overthink it — start simple and add detail if the output isn't what you want.

You don't need all five components every time. The formula exists for when results are bad, not as a mandatory ritual. Day-to-day use is conversational.

The highest-leverage thing you can do

Before you even think about prompting technique: be clear about your goal. AI amplifies direction. If your goal is fuzzy, the output will be fuzzy. If your goal is precise, the output is dramatically better.

This applies at every level. A session goal ("implement step-completion API with tests") is better than a vague one ("work on the onboarding wizard"). A clear problem statement ("rate limiting for a specific abusive client") beats a generic one ("fix performance").

Goal clarity isn't just an AI skill — it makes your PRs better, your tickets clearer, and your design docs sharper. It just happens to be the single highest-leverage thing you can do to improve AI output.

6. Context Is Everything

If there's one concept that ties everything in this guide together, it's this: the quality of Claude's output is directly proportional to the quality of context you give it. Bad output is almost always a context problem, not an intelligence problem.

Why longer conversations get worse

Remember that the entire conversation history rides with every message. By message 50, Claude is processing tens of thousands of tokens of old conversation — including dead ends, discarded ideas, and outdated instructions. The signal-to-noise ratio degrades. And because of the "lost in the middle" problem, important context from 30 messages ago might as well not exist.

This is also why long conversations are expensive. A fresh message costs a handful of tokens. The same message at turn 50 can cost 10–50x more, because it's carrying all that history.

Key commands

CommandWhat it doesWhen to use it
/clearWipes conversation and starts freshSwitching tasks, context gone off the rails
/planSwitches to plan mode — Claude explores and proposes before writing codeComplex tasks, unfamiliar code, before context gets heavy
/statsShows session usage patterns and how close you are to limitsChecking if a conversation is getting expensive
/resumePicks up a previous conversation where you left offReturning to work after a break or a /clear
/exitGracefully ends the Claude Code sessionDone working, or switching to a different project

The most powerful technique: plan in one conversation, execute in a new one. Use one session to explore, discuss, and make decisions. Then start a fresh conversation for the actual implementation. The planning conversation is full of dead ends and exploration. The execution conversation is clean and focused.

Context hygiene as a habit

Think of context management the way you think about git hygiene. You don't let branches live forever. You don't commit everything to main. You keep things clean because clean state produces better work.

Same principle: start fresh conversations often. Don't be precious about conversation history. Use /clear when switching tasks. Use /stats to check how heavy your session is getting. And when you're doing anything complex, plan first, then start fresh for execution.

Limits are a diagnostic

If you're hitting rate limits regularly, that's almost always a context management problem, not a subscription problem. It means your conversations are running too long and each message is consuming too many tokens. The fix isn't a bigger plan — it's shorter, more focused conversations.

You can check how close you are to your limits anytime with /stats.

7. CLAUDE.md & Project Memory

We just said AI starts blank every time. So how do you avoid repeating yourself in every conversation? The answer is CLAUDE.md — a markdown file that Claude reads at the start of every session.

CLAUDE.md — A markdown file that Claude reads automatically at the start of every conversation. You can place them at multiple levels — your home directory for personal preferences, your repo root for project conventions, and subdirectories for team-specific patterns. Claude reads them hierarchically as it navigates your code. Think of it as your project's layered brain.

The hierarchy

~/.claude/CLAUDE.md            ← Your personal preferences (applies everywhere)
your-repo/
├── CLAUDE.md                  ← Project-level rules and conventions
├── frontend/
│   └── CLAUDE.md              ← Frontend-specific patterns
├── backend/
│   └── CLAUDE.md              ← Backend-specific conventions
└── docs/
    ├── architecture.md        ← Deep reference docs
    ├── glossary.md            ← Domain terminology
    └── api-contracts.md       ← API specifications

Claude reads these hierarchically — root first, then deeper as it navigates into your code. The home-level file (~/.claude/CLAUDE.md) applies to every project. The repo-level file applies to that specific codebase.

What goes in CLAUDE.md

Think of it as instructions for a smart new teammate on their first day:

  • Architecture: "This is a Rails 7 API with an Angular frontend in a monorepo managed by NX"
  • Conventions: "We use the interactor pattern for business logic. Services live in app/services/."
  • Standards: "Always use safe-pg-migrations for database changes. Never use before_filter."
  • Testing: "Write request specs for API endpoints. Use FactoryBot for test data."
  • Terminology: "A 'tracker' is a phone number pool used for call attribution."

Keep it concise. CLAUDE.md gets loaded every conversation, so every line costs tokens. Link to deeper docs in a docs/ folder rather than inlining everything.

Documentation as code

Here's the paradigm shift: your documentation isn't just for humans anymore. When Claude can read your architecture docs, domain glossary, and API contracts, it writes code that fits your system instead of guessing.

Markdown files in a docs/ folder, version controlled with git, are the simplest and most effective way to give Claude (and your team) persistent project knowledge. No special tools needed — just folders and files.

/init — A Claude Code command that reads your codebase and generates a starter CLAUDE.md. It examines your project structure, README, configs, and dependencies to create a first draft. The real value comes from human curation — the generated file is a starting point, not the finished product.
Advanced: The token cost of CLAUDE.md

Every line of CLAUDE.md gets loaded into the context window at the start of every conversation — and it rides along with every single message. A 500-line CLAUDE.md might be 2,000+ tokens. Over a 30-message conversation, that's 60,000 tokens just for CLAUDE.md alone.

This creates a tradeoff: more instructions means better-informed Claude, but also means less room for actual work and higher costs per message. Best practices:

  • Keep CLAUDE.md under ~200 lines. Enough for critical conventions, not a novel.
  • Use docs/ for deep reference. Put your architecture docs, glossary, and API contracts in separate files. Claude reads these on-demand when it navigates into that code — they don't load every conversation.
  • Subdirectory CLAUDE.md files only load when relevant. A frontend/CLAUDE.md only enters the context when Claude is working in the frontend directory. Use this to scope heavy instructions.

8. Plan Mode

When you give Claude a task in normal mode, it starts working immediately — reading files, writing code, running commands. That's great for simple, well-defined tasks. But for anything complex, you want Claude to think before it builds.

That's plan mode.

Plan mode — A mode where Claude explores your codebase and proposes a plan before making any changes. It reads, analyzes, and presents a strategy for your approval. No code is written until you approve the plan. Enter with Shift+Tab or by typing "plan" in your message.

When to use it

Plan ModeNormal Mode
Claude doesReads, explores, proposes planReads, writes, executes
You doReview strategy, adjust, approveWatch, steer, approve actions
Best forMulti-file changes, unfamiliar code, complex tasksSimple fixes, well-defined changes

The power of plan mode is that before a single line of code is written, you've reviewed the entire strategy. You can catch wrong assumptions, redirect the approach, or refine the scope — when it's cheap to change, not after 500 lines of code are already written.

Plan, then start fresh

One of the most powerful patterns in Claude Code: use plan mode in one conversation to explore and make decisions. Then start a fresh conversation for execution. The planning conversation is full of exploration and dead ends. The execution conversation is clean, focused, and carries only the final plan — not all the noise that led to it.

When context runs high, switch to plan mode

Here's a tactical move that separates experienced users from beginners: when you notice your context is getting heavy — you've been working for a while, Claude has read lots of files, the conversation is getting long — switch to plan mode for your next step. Use your remaining context to have Claude lay out a clear plan for what comes next. Then start a fresh conversation to execute that plan with a clean, focused context window.

This turns a potential problem (high context usage) into a strategic advantage. Instead of pushing through with degrading quality, you capture Claude's full understanding of your session into a concrete plan, then bring that plan into a fresh session where every token goes toward execution instead of carrying dead weight.

9. Steering & Summoning

There are two fundamentally different ways to work with Claude Code: steering and summoning. Understanding the difference — and when to use each — is one of the most important skills you'll develop.

Steering: you're in the loop

Steering means working alongside Claude in real time. You watch what it does, correct course when it drifts, and provide guidance as it works.

  • "Use Tailwind for that, not inline styles"
  • "Stop — just the controller, not the model"
  • "Good, but add error handling for the timeout case"

Steering is slower but produces better results. You learn while Claude works. And critically, every correction you make is an opportunity to add that guidance to CLAUDE.md so it sticks for future sessions.

Summoning: Claude works autonomously

Summoning — Triggering Claude to work autonomously without real-time steering. Mention @claude on a GitHub issue and it implements and opens a PR. Mention it on a PR and it reviews or fixes. Use /claude in Slack for quick tasks. Fast, but you get what you get.

Summoning is powerful for well-defined, low-risk tasks. Claude gets a task, does the work, and delivers the result — without you watching over its shoulder.

The 80/20 reality

Claude gets it right about 80% of the time. Steering closes the gap on the other 20%. For important work — features that matter, code that's tricky, areas you're unfamiliar with — steer. For well-defined, lower-stakes tasks — doc updates, boilerplate, simple fixes — summon.

Over time, as your CLAUDE.md improves and Claude has better context about your project, the 80% gets closer to 90%. Every steering correction you capture in your documentation makes all future sessions better.

10. Commands, Skills & Extending

Everything so far has been about Claude's built-in capabilities. But the real power comes when you start customizing Claude for your specific workflows.

Commands

A command is a prompt template saved as a markdown file in .claude/commands/ in your repo. It becomes a slash command anyone on your team can trigger manually.

Command — A reusable prompt template saved as a markdown file in .claude/commands/. Becomes a slash command (like /deploy-check or /prime-context) that you invoke explicitly.

For example, a /prime-context command might read your key architecture docs, recent PRs, and open issues — loading all your project context at the start of a session. A /deploy-check command might run your test suite, verify the build, and open a PR. One command, full workflow.

Skills

Skills live in .claude/skills/ and work differently from commands: Claude finds and uses them automatically when they're relevant. You don't need to invoke them with a slash command — Claude reads the skill descriptions and decides when to apply them on its own.

Skill — A markdown file in .claude/skills/ that Claude discovers and uses automatically when relevant. Unlike commands, you don't trigger them — Claude reads the description and decides when to apply the instructions.

You can create both by hand (they're just markdown) or by running claude create skill or claude create command, which interviews you about your workflow and generates the file.

Because commands and skills live in your repo, they're version controlled. When someone improves one, the whole team gets the improvement on their next pull.

CLI tools: Claude already knows them

Claude has been trained on millions of man pages. It already knows how to use git, gh, awscli, kubectl, helm, terraform, rails, nx, docker, and most other public CLI tools. If you can do it in your terminal, Claude can probably do it too.

For internal tools it doesn't know, just describe them in CLAUDE.md and it picks them up.

MCP: when CLI isn't enough

MCP (Model Context Protocol) — A plugin system that gives Claude structured access to external tools and services. Used when there's no CLI option — things like Figma, design tools, or specialized databases.

The general rule: default to CLI tools, use MCP when there's no CLI option. CLI tools are simpler, more debuggable, and Claude already knows them. MCP is powerful but adds complexity.

Hooks

Hooks — Your code that runs automatically before or after Claude takes actions. Useful for things like sending a notification when Claude finishes a long task, or running a linter before commits.

Hooks solve a common problem: you kick off a task, switch to something else, and forget to check back. A hook can ping you in Slack when Claude's done.

Built-in subagents

Claude Code doesn't just work alone — it can spawn subagents to handle parts of a task in parallel. These are lightweight Claude instances that run inside your main session, each focused on a specific job.

Two built-in subagent types you'll see frequently:

  • Research agents — Claude spawns these to explore your codebase or search the web without cluttering your main conversation. They read files, search for patterns, and report back with findings. Useful when Claude needs to understand something before making changes.
  • Task agents — These handle discrete pieces of work autonomously. Claude might spawn several in parallel to tackle independent subtasks — running tests, checking multiple files, or implementing separate components simultaneously.

You don't need to configure subagents — Claude decides when to use them based on the complexity of your request. But knowing they exist helps you understand why Claude sometimes pauses to "think" before acting: it's coordinating work across multiple agents behind the scenes.

11. When Things Go Wrong

AI tools have outages, rate limits, and bad days — just like any other service. Having a plan for when things go sideways keeps you productive instead of frustrated.

Rate limits

Claude's subscription has two limit layers: a 5-hour rolling window that caps burst usage, and a weekly cap on total compute time. When you hit a limit, you'll get a clear error message telling you when it resets.

If you're hitting limits regularly, remember: that's usually a context management signal, not a subscription problem. Long conversations burn tokens fast. Use /stats to check your session usage, and /clear to start fresh when switching tasks.

Outages and degraded performance

Sometimes Claude is fully down; sometimes it's up but struggling — slow responses, timeouts, or noticeably worse output quality. Either way, check status.anthropic.com to see what's going on. When Claude isn't performing well, continuing to work wastes tokens on bad results.

When you hit limits or Claude is down

Switch to another agentic coder. Claude Code isn't the only game in town. When you're rate-limited or dealing with an outage, try one of these:

  • OpenAI Codex CLI — OpenAI's agentic coding tool. Works similarly to Claude Code but runs on GPT models. Good fallback when Claude is unavailable.
  • Gemini CLI — Google's agentic CLI. Runs on Gemini models and has a generous free tier. Worth having installed as a backup.
  • Cursor / Copilot — IDE-based alternatives if you prefer a visual environment while waiting.

Having at least one alternative installed and configured means an outage is a minor inconvenience, not a blocked afternoon. The skills you're learning in this workshop — clear prompts, context management, CLAUDE.md — transfer directly to other agentic tools.

When none of the AI tools are working for you, use the downtime productively:

  • Do code review — the kind that benefits from human attention
  • Update your CLAUDE.md and docs — make future sessions better
  • Write handoff notes for your next Claude session
  • Plan upcoming work

Terms worth knowing

Frontier model
The most capable models pushing the state of the art. Claude Opus 4.6, GPT-5.4, and Gemini 3.1 Pro are today's frontier models — and for the first time, they're within a percentage point of each other on major coding benchmarks. These are what you want for complex engineering tasks.
Open-weight / open-source
Models whose weights are publicly available — Meta's Llama 4 family is the leader, with Mistral also notable. You can run them locally or host them yourself. Generally less capable than frontier models but free and private. "Open-weight" means the weights are public but the training data and process aren't — an important distinction.
Parameters
The "size" of a model — the number of learned values in its neural network. Llama 3 comes in 8B and 70B parameter versions. More parameters generally means more capable but slower and more expensive. Frontier model sizes are typically not disclosed.
Inference
Running a trained model to get output — as opposed to training it. Every time you send Claude a message, that's inference. When people talk about "inference costs," they mean the cost of generating responses (tokens out).
Fine-tuning
Taking a pre-trained model and training it further on specific data to specialize it. Common with open-weight models. Not something you do with Claude or ChatGPT directly — instead, you steer them with prompts and context.

The landscape changes fast, but the fundamentals — tokens, context, prompting, agentic workflows — are stable across all of these. What you learn with Claude transfers.

Glossary

Agentic
An AI that takes actions autonomously — reads files, runs commands, writes code, iterates on results — rather than just answering questions.
Chat / Cowork / Code
The three modes in the Claude Desktop app. Chat is for quick conversations. Cowork handles sustained knowledge work with access to shared folders. Code is a full development environment with terminal access. Cowork and Code run on the same engine.
CLI (Command Line Interface)
Claude Code runs in your terminal, giving it access to your filesystem, git, and every CLI tool you have installed.
CLAUDE.md
Markdown files Claude reads automatically — placed at your home directory, repo root, and/or subdirectories. They stack hierarchically to give Claude layered project memory.
Context
Everything the AI can "see" during a conversation: your messages, its responses, files it's read, command outputs.
Context window
The total working memory for a single conversation. Claude's is ~200,000 tokens today (1M announced). What's in the window matters more than how big it is.
Hooks
Your code that runs automatically before or after Claude takes actions — notifications, linting, validation.
MCP (Model Context Protocol)
A plugin system for giving Claude structured access to external tools and services. Use when CLI isn't an option.
Modality
A type of data a model can work with — text, images, audio, video. Models that handle multiple types are called multimodal.
Model
The AI brain itself. Claude comes in Haiku (fast/cheap), Sonnet (balanced), and Opus (most capable — recommended for daily engineering work).
Multimodal
A model that can process more than just text. Claude handles text + images + PDFs. Other models add audio, video, and image generation.
Plan mode
A mode where Claude explores and proposes a strategy before making changes. Enter with Shift+Tab or by typing "plan."
Prompt
The message you send to AI. Can be conversational or structured. Better context produces better results.
Command
A reusable prompt template saved in .claude/commands/. You trigger it manually with a slash command (e.g. /prime-context).
Skill
A markdown file in .claude/skills/ that Claude discovers and uses automatically when relevant — no slash command needed.
Steering
Working alongside Claude in real time — watching, correcting course, providing guidance as it works.
Summoning
Triggering Claude to work autonomously via GitHub @mentions, Slack commands, or other integrations.
Token
A subword unit (~4 characters of English text) created by Byte Pair Encoding. The native unit of all LLM input and output. Common words = 1 token; rare words split into pieces. Code, non-English text, and encoded strings tokenize less efficiently.
/clear
Wipes conversation history and starts a fresh session. Use when switching tasks or when context is polluted.
/plan
Switches to plan mode where Claude explores and proposes a strategy before making changes. Use Shift+Tab as a shortcut.
/stats
Shows your current session's usage patterns and how close you are to your limits. Useful for knowing when to /clear or switch to plan mode.
/init
Reads your codebase and generates a starter CLAUDE.md. A first draft — the real value comes from human curation.
