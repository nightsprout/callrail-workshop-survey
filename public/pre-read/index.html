1. How AI Actually Works

Every interaction with AI runs on the same basic loop: you send text in, the model processes it, and text comes back. Understanding what's happening under the hood helps you debug problems and get better results.

It starts with tokens

AI doesn't read words the way you do. It breaks everything down into tokens — fragments of text, roughly three-quarters of a word each. The word "development" is 2 tokens. "CallRail" is 2 tokens. A short paragraph might be 50 tokens. A full source file could be thousands.

Token — The smallest unit of text AI processes. Roughly 3/4 of a word. Everything you send, everything AI reads, everything it generates — all measured in tokens.

Tokens matter because they're what you're paying for. Every file Claude reads, every command output it processes, every response it writes — all tokens. A long conversation isn't just slower, it's literally more expensive per message, because the entire conversation history rides along with every new message you send.

AI starts blank every time

This is the single most important thing to understand. Claude doesn't remember your last conversation. It doesn't know what you worked on yesterday. Every new session starts from zero — a blank slate with no memory of anything that came before.

Everything Claude knows in a given conversation comes from what's been injected into that conversation's context window — the working memory for that session.

Context window — The total amount of text AI can hold in working memory for a single conversation. Claude's is roughly 200,000 tokens. Bigger isn't always better — what matters is what's in the window.

Your prompt, the conversation history, any files Claude reads, any command outputs — it all lives in the context window. When it fills up, older content starts getting compressed or dropped. And here's the catch: AI is best at remembering what's at the beginning and end of the context window. Stuff in the middle gets hazy — researchers call this the "lost in the middle" problem.

Context — Everything the AI can "see" during a conversation: your messages, its responses, files it's read, command outputs. More context isn't always better — focused, relevant context beats a massive dump.

Generation is prediction

When Claude writes a response, it's not retrieving answers from a database. It's predicting the next token, one at a time, based on patterns learned during training. This is why the same prompt can produce different outputs — it's probabilistic, not deterministic. It's also why giving Claude better context produces better outputs. More signal, less noise, better predictions.

That's the whole loop: tokens in → context window → prediction out. Everything else we'll cover — managing context, CLAUDE.md, plan mode, steering — is about optimizing this loop.

Chat is what you'd expect — the same Claude from claude.ai, but running natively on your machine. Double-tap Option (Mac) to pull it up over any window for quick questions.

Cowork is for work that takes real effort. Give Claude a folder and a task — "research these three competitors and write a summary" or "analyze this contract and flag risks" — and it reads what's there, works through it, and saves finished output back. It can even run on a schedule for recurring tasks like daily briefings. Think of it as a capable research assistant with access to your files.

Code is the same Claude Code engine that runs in your terminal, but inside the desktop app with visual diffs and a built-in terminal. It has three interaction sub-modes:

  • Ask: Claude proposes every change with a visual diff — you review and approve before anything is modified
  • Code: Changes apply automatically, but terminal commands still require approval
  • Plan: Claude outlines a strategy before executing anything (same concept as plan mode in the CLI)

The important thing to understand: Cowork and Code run on the same engine underneath. Both can work independently, spin up sub-agents for parallel tasks, and sustain long-running work. The difference is scope — Code has your full project and terminal, Cowork is limited to the folders you share.

In the workshop, we'll focus on Claude Code (the CLI), but everything you learn about prompting, context management, and CLAUDE.md applies across all three modes.

3. What Makes Claude Code Different

You may have used GitHub Copilot, or ChatGPT, or even claude.ai for coding questions. Claude Code is fundamentally different from all of these. Understanding how it's different is the key to using it well.

The spectrum

Think of AI coding tools on a spectrum:

Autocomplete → Copilot → Chat → Agentic

Autocomplete suggests the next line. Copilot suggests blocks of code as you type. Chat tools answer questions about code. Agentic tools do the work.

Agentic — An AI that doesn't just answer questions — it takes actions. It reads your files, runs commands, creates branches, writes code, runs tests, and iterates on its own work. You describe what you want; it figures out how to get there.

Claude Code sits at the far end of that spectrum. It's not suggesting code in your editor — it's operating your entire development environment. It can:

  • Read and write files across your whole project
  • Run shell commands (build, test, lint, deploy)
  • Search your codebase for patterns and dependencies
  • Create branches, make commits, open pull requests
  • Plan multi-step changes before executing them
  • Spawn sub-agents to research or work in parallel

It asks permission before making changes (by default), so you're always in control. But the scope of what it can do is fundamentally different from autocomplete — it's closer to a junior developer with perfect memory of your codebase than it is to a fancy text completer.

Where it lives

Claude Code runs in your terminal. Not in a browser tab, not in an IDE sidebar — in the same terminal where you run git, rails, npm, and everything else. This is intentional: it has access to the same tools you do.

There's also a desktop app for more visual conversations, and integrations with GitHub (for PR review) and Slack (for team use). But the core experience is the CLI.

CLI — Command Line Interface. Claude Code runs in your terminal alongside your other dev tools. This gives it access to your filesystem, git, and every CLI tool you have installed.

4. Talking to Claude

Good news: you don't need to learn a special language. Most of the time, you talk to Claude the way you'd talk to a colleague. "Fix the failing test in user_spec.rb." "Add pagination to the trackers endpoint." "What does this service do?"

That said, when you're not getting the results you want, there's a formula that helps.

The prompting formula

ComponentWhat it doesExample
GoalWhat you're trying to accomplish"I need to add rate limiting to our API"
RoleWhat perspective Claude should take"Act as a senior Rails engineer familiar with our stack"
ProblemWhat's broken, missing, or needed"We're getting hammered by a single client doing 1000 req/min"
DetailsConstraints and context"We use Redis, Rails 7, and rack-attack is already in the Gemfile"
ExamplesWhat good output looks like"Similar to how we rate-limit the webhooks controller"
Prompt — The message you send to AI. Can be a simple question or a detailed request. Better prompts produce better results, but don't overthink it — start simple and add detail if the output isn't what you want.

You don't need all five components every time. The formula exists for when results are bad, not as a mandatory ritual. Day-to-day use is conversational.

The highest-leverage thing you can do

Before you even think about prompting technique: be clear about your goal. AI amplifies direction. If your goal is fuzzy, the output will be fuzzy. If your goal is precise, the output is dramatically better.

This applies at every level. A session goal ("implement step-completion API with tests") is better than a vague one ("work on the onboarding wizard"). A clear problem statement ("rate limiting for a specific abusive client") beats a generic one ("fix performance").

Goal clarity isn't just an AI skill — it makes your PRs better, your tickets clearer, and your design docs sharper. It just happens to be the single highest-leverage thing you can do to improve AI output.

5. Context Is Everything

If there's one concept that ties everything in this guide together, it's this: the quality of Claude's output is directly proportional to the quality of context you give it. Bad output is almost always a context problem, not an intelligence problem.

Why longer conversations get worse

Remember that the entire conversation history rides with every message. By message 50, Claude is processing tens of thousands of tokens of old conversation — including dead ends, discarded ideas, and outdated instructions. The signal-to-noise ratio degrades. And because of the "lost in the middle" problem, important context from 30 messages ago might as well not exist.

This is also why long conversations are expensive. A fresh message costs a handful of tokens. The same message at turn 50 can cost 10–50x more, because it's carrying all that history.

Key commands

CommandWhat it doesWhen to use it
/clearWipes conversation and starts freshSwitching tasks, context gone off the rails
/planSwitches to plan mode — Claude explores and proposes before writing codeComplex tasks, unfamiliar code, before context gets heavy
/statsShows session usage patterns and how close you are to limitsChecking if a conversation is getting expensive
/resumePicks up a previous conversation where you left offReturning to work after a break or a /clear

The most powerful technique: plan in one conversation, execute in a new one. Use one session to explore, discuss, and make decisions. Then start a fresh conversation for the actual implementation. The planning conversation is full of dead ends and exploration. The execution conversation is clean and focused.

Context hygiene as a habit

Think of context management the way you think about git hygiene. You don't let branches live forever. You don't commit everything to main. You keep things clean because clean state produces better work.

Same principle: start fresh conversations often. Don't be precious about conversation history. Use /clear when switching tasks. Use /stats to check how heavy your session is getting. And when you're doing anything complex, plan first, then start fresh for execution.

Limits are a diagnostic

If you're hitting rate limits regularly, that's almost always a context management problem, not a subscription problem. It means your conversations are running too long and each message is consuming too many tokens. The fix isn't a bigger plan — it's shorter, more focused conversations.

You can check how close you are to your limits anytime with /stats.

6. CLAUDE.md & Project Memory

We just said AI starts blank every time. So how do you avoid repeating yourself in every conversation? The answer is CLAUDE.md — a markdown file that Claude reads at the start of every session.

CLAUDE.md — A markdown file at the root of your project that Claude reads automatically at the start of every conversation. It contains project conventions, architecture notes, common patterns, and anything else Claude should know. Think of it as your project's brain.

The hierarchy

~/.claude/CLAUDE.md            ← Your personal preferences (applies everywhere)
your-repo/
├── CLAUDE.md                  ← Project-level rules and conventions
├── frontend/
│   └── CLAUDE.md              ← Frontend-specific patterns
├── backend/
│   └── CLAUDE.md              ← Backend-specific conventions
└── docs/
    ├── architecture.md        ← Deep reference docs
    ├── glossary.md            ← Domain terminology
    └── api-contracts.md       ← API specifications

Claude reads these hierarchically — root first, then deeper as it navigates into your code. The home-level file (~/.claude/CLAUDE.md) applies to every project. The repo-level file applies to that specific codebase.

What goes in CLAUDE.md

Think of it as instructions for a smart new teammate on their first day:

  • Architecture: "This is a Rails 7 API with an Angular frontend in a monorepo managed by NX"
  • Conventions: "We use the interactor pattern for business logic. Services live in app/services/."
  • Standards: "Always use safe-pg-migrations for database changes. Never use before_filter."
  • Testing: "Write request specs for API endpoints. Use FactoryBot for test data."
  • Terminology: "A 'tracker' is a phone number pool used for call attribution."

Keep it concise. CLAUDE.md gets loaded every conversation, so every line costs tokens. Link to deeper docs in a docs/ folder rather than inlining everything.

Documentation as code

Here's the paradigm shift: your documentation isn't just for humans anymore. When Claude can read your architecture docs, domain glossary, and API contracts, it writes code that fits your system instead of guessing.

Markdown files in a docs/ folder, version controlled with git, are the simplest and most effective way to give Claude (and your team) persistent project knowledge. No special tools needed — just folders and files.

/init — A Claude Code command that reads your codebase and generates a starter CLAUDE.md. It examines your project structure, README, configs, and dependencies to create a first draft. The real value comes from human curation — the generated file is a starting point, not the finished product.

7. Plan Mode

When you give Claude a task in normal mode, it starts working immediately — reading files, writing code, running commands. That's great for simple, well-defined tasks. But for anything complex, you want Claude to think before it builds.

That's plan mode.

Plan mode — A mode where Claude explores your codebase and proposes a plan before making any changes. It reads, analyzes, and presents a strategy for your approval. No code is written until you approve the plan. Enter with Shift+Tab or by typing "plan" in your message.

When to use it

Plan ModeNormal Mode
Claude doesReads, explores, proposes planReads, writes, executes
You doReview strategy, adjust, approveWatch, steer, approve actions
Best forMulti-file changes, unfamiliar code, complex tasksSimple fixes, well-defined changes

The power of plan mode is that before a single line of code is written, you've reviewed the entire strategy. You can catch wrong assumptions, redirect the approach, or refine the scope — when it's cheap to change, not after 500 lines of code are already written.

Plan, then start fresh

One of the most powerful patterns in Claude Code: use plan mode in one conversation to explore and make decisions. Then start a fresh conversation for execution. The planning conversation is full of exploration and dead ends. The execution conversation is clean, focused, and carries only the final plan — not all the noise that led to it.

When context runs high, switch to plan mode

Here's a tactical move that separates experienced users from beginners: when you notice your context is getting heavy — you've been working for a while, Claude has read lots of files, the conversation is getting long — switch to plan mode for your next step. Use your remaining context to have Claude lay out a clear plan for what comes next. Then start a fresh conversation to execute that plan with a clean, focused context window.

This turns a potential problem (high context usage) into a strategic advantage. Instead of pushing through with degrading quality, you capture Claude's full understanding of your session into a concrete plan, then bring that plan into a fresh session where every token goes toward execution instead of carrying dead weight.

8. Steering & Summoning

There are two fundamentally different ways to work with Claude Code: steering and summoning. Understanding the difference — and when to use each — is one of the most important skills you'll develop.

Steering: you're in the loop

Steering means working alongside Claude in real time. You watch what it does, correct course when it drifts, and provide guidance as it works.

  • "Use Tailwind for that, not inline styles"
  • "Stop — just the controller, not the model"
  • "Good, but add error handling for the timeout case"

Steering is slower but produces better results. You learn while Claude works. And critically, every correction you make is an opportunity to add that guidance to CLAUDE.md so it sticks for future sessions.

Summoning: Claude works autonomously

Summoning — Triggering Claude to work autonomously without real-time steering. Mention @claude on a GitHub issue and it implements and opens a PR. Mention it on a PR and it reviews or fixes. Use /claude in Slack for quick tasks. Fast, but you get what you get.

Summoning is powerful for well-defined, low-risk tasks. Claude gets a task, does the work, and delivers the result — without you watching over its shoulder.

The 80/20 reality

Claude gets it right about 80% of the time. Steering closes the gap on the other 20%. For important work — features that matter, code that's tricky, areas you're unfamiliar with — steer. For well-defined, lower-stakes tasks — doc updates, boilerplate, simple fixes — summon.

Over time, as your CLAUDE.md improves and Claude has better context about your project, the 80% gets closer to 90%. Every steering correction you capture in your documentation makes all future sessions better.

9. Skills, Commands & Extending

Everything so far has been about Claude's built-in capabilities. But the real power comes when you start customizing Claude for your specific workflows.

Skills and commands

A skill is just a markdown file with instructions. You put it in .claude/commands/ or .claude/skills/ in your repo, and it becomes a slash command anyone on your team can use.

Skill / Command — A reusable set of instructions saved as a markdown file in your repo's .claude/ folder. Once saved, it becomes a slash command (like /deploy-check) that anyone on the team can run.

For example, a /do skill might tell Claude to: stash uncommitted changes, create a new branch, implement a task, verify it works, then commit and open a PR. One command, full workflow. A /prime skill might load all your project context at the start of a session.

You can create skills by hand (they're just markdown) or by running claude create skill, which interviews you about your workflow and generates the file.

Because skills live in your repo, they're version controlled. When someone improves a skill, the whole team gets the improvement on their next pull.

CLI tools: Claude already knows them

Claude has been trained on millions of man pages. It already knows how to use git, gh, awscli, kubectl, helm, terraform, rails, nx, docker, and most other public CLI tools. If you can do it in your terminal, Claude can probably do it too.

For internal tools it doesn't know, just describe them in CLAUDE.md and it picks them up.

MCP: when CLI isn't enough

MCP (Model Context Protocol) — A plugin system that gives Claude structured access to external tools and services. Used when there's no CLI option — things like Figma, design tools, or specialized databases.

The general rule: default to CLI tools, use MCP when there's no CLI option. CLI tools are simpler, more debuggable, and Claude already knows them. MCP is powerful but adds complexity.

Hooks

Hooks — Your code that runs automatically before or after Claude takes actions. Useful for things like sending a notification when Claude finishes a long task, or running a linter before commits.

Hooks solve a common problem: you kick off a task, switch to something else, and forget to check back. A hook can ping you in Slack when Claude's done.

10. When Things Go Wrong

AI tools have outages, rate limits, and bad days — just like any other service. Having a plan for when things go sideways keeps you productive instead of frustrated.

Rate limits

Claude's subscription has two limit layers: a 5-hour rolling window that caps burst usage, and a weekly cap on total compute time. When you hit a limit, you'll get a clear error message telling you when it resets.

If you're hitting limits regularly, remember: that's usually a context management signal, not a subscription problem. Long conversations burn tokens fast. Use /stats to check your session usage, and /clear to start fresh when switching tasks.

Degraded performance

Sometimes Claude is up but struggling — slow responses, timeouts, or noticeably worse output quality. When this happens, continuing to work wastes tokens on bad results. It's better to switch tools temporarily or do something else.

Full outages

Check status.anthropic.com to confirm. If Claude is down, you're down — unless you have a backup plan.

The redirect playbook

When Claude isn't available or isn't performing well, use the downtime productively:

  • Switch to an alternative AI tool (Gemini, Copilot, Cursor) for immediate needs
  • Do code review — the kind that benefits from human attention
  • Update your CLAUDE.md and docs — make future sessions better
  • Write handoff notes for your next Claude session
  • Run manual testing
  • Plan upcoming work

The engineers who stay most productive with AI are the ones who planned for this moment before it happened.
